## 1. 制作意図

- **メンバー・活動内容を継続的に発信できる**（ブログ／活動記録）
- **プログラミングに不慣れな運営メンバーでも記事を投稿・編集できる**（管理画面とリッチエディタ）
- **無料の範囲で安定運用できる**（個人運営のサークルなので運用コストをかけられない）

という3点を満たすサイトを目標に設計した。「作って終わり」ではなく、**実際に運用に乗ること**を最優先の要件として、技術選定・インフラ構成・コスト設計まで一貫して自分で判断した。

## 3. 技術スタック

| レイヤ | 技術 |
|---|---|
| フロントエンド | React 19, TypeScript, Vite, Chakra UI, React Router v7 |
| データ通信 | Apollo Client, GraphQL, **GraphQL Code Generator**（スキーマからの型自動生成） |
| リッチエディタ | TipTap v3（独自拡張：スラッシュコマンド／メンション／数式／テンプレート） + KaTeX |
| バックエンド | NestJS 11, TypeScript, **GraphQL（code-first）**, Apollo Server v5 |
| DB / ORM | PostgreSQL, Prisma |
| 認証 | JWT + Passport（passport-jwt）, bcrypt |
| メール送信 | Nodemailer（お問い合わせフォーム） |
| インフラ | フロント: GitHub Pages ／ API: Render ／ DB: Neon（いずれも無料枠） |
| CI/CD・運用 | GitHub Actions（push時の自動デプロイ）／ cron-job.org（APIの死活監視・keep-warm） |

### アーキテクチャ

```
[ ユーザーのブラウザ ]
        │  GraphQL (HTTPS)
        ▼
[ GitHub Pages ]  ← React/TypeScript（静的ホスティング）
        │  Apollo Client
        ▼
[ Render ]        ← NestJS + Apollo Server（GraphQL API）
        │  Prisma
        ▼
[ Neon ]          ← PostgreSQL

[ GitHub Actions ] ── push時に自動ビルド&デプロイ
[ cron-job.org ]   ── 10分毎にAPIを起こすkeep-warm（後述）
```

ソースコード規模: フロント 約2,400行 / バックエンド 約800行（自動生成コードを除く、TypeScript）。

## 4. 主な機能

- **メンバー紹介** … 役職・学年・自己紹介・SNSリンクを一覧表示
- **ブログ / 活動記録** … 記事の一覧・詳細表示（2種類の投稿タイプを区別）
- **管理者用 記事エディタ** … 後述の自作リッチテキストエディタで記事を作成・編集・テンプレート適用
- **管理者認証** … JWTによるログイン。**編集系の操作は管理者のみに制限**
- **お問い合わせフォーム** … 入力内容をサーバ経由でメール送信（Nodemailer）
- **レスポンシブ対応** … スマホ閲覧時のナビゲーション挙動やフォーム入力（iOSの自動ズーム抑止など）まで個別調整

## 5. 技術的に工夫した点

選考で見ていただく上で、特に「自分で考えて判断した」部分を挙げる。

### (1) 無料インフラの制約を理解した上でのコスト設計と運用改善 🔧

サークル運営なので**運用コストをかけられない**という制約から、フロント(GitHub Pages)・API(Render)・DB(Neon)をすべて無料枠で構成した。しかし無料プランには固有の制約があり、それぞれを調べた上で設計判断をしている。

- **課題**: Render無料プランは15分アクセスがないとサーバがスリープし、復帰に約40秒かかる（実測42秒）。サイトを開くたびに最初のリクエストが極端に遅い。
- **さらに**: 単純に「定期的に叩いて起こし続ける」と、今度は**DB(Neon)の無料枠（コンピュート100時間/月）を食い潰す**ことに気づいた。Neonは5分で自動停止するため、DBに触れるリクエストで起こし続けると、月の途中で枠を超えてDBごと停止してしまう。
- **解決**: 外部の死活監視サービス（cron-job.org）から10分毎に **DBに触れないクエリ（`{ __typename }`）** だけをPOSTする keep-warm を構築した。これにより **「APIサーバは起こし続けつつ、DBは眠らせる」** ことを実現し、両方の無料枠を破綻させずにコールドスタートを解消した。
  - 当初はGitHub Actionsのスケジュール実行で組んだが、無料の `schedule` は遅延・実行漏れが多く「15分以内に必ず1回」を保証できなかった。そこで短間隔の定期実行に特化したcron-job.orgへ切り替え、ツール選定まで含めて目的（確実な死活維持）から逆算して判断した。

> ボトルネックの切り分け（ネットワークかサーバ起動か）を実測で行い、各サービスの課金単位（稼働時間・CU時間）を理解した上で最適点を見つけた、という一連の問題解決である。**CI/CDやビルド効率化の発想**に近い取り組みだと考えている。

### (2) スキーマ駆動の型安全な設計 🧩

バックエンドのGraphQLスキーマ（NestJSのcode-first）を単一の真実の情報源とし、**GraphQL Code Generator** でフロントエンドのTypeScript型を自動生成している。

- APIのレスポンス型を手書きせず、**バックエンドの定義変更がフロントの型に自動で反映**される。
- クエリの書き間違いやフィールドの型ミスを**コンパイル時に検出**できる。
- 「バックエンドを直したのにフロントの型を直し忘れる」という典型的な不整合を構造的に防止できる。

一人開発でも変更に強く壊れにくい構成を、ツールチェインのレベルで担保した。

### (3) 自作リッチテキストエディタ ✍️

運営メンバーが直感的に記事を書けるよう、TipTapを拡張した独自エディタを実装した。

- **スラッシュコマンド**（`/`で見出し・リスト等を挿入するNotion風UI）
- **メンション**、**数式入力（KaTeX）**、**テンプレート挿入**（定型記事の雛形をDBから読み込み）

外部のCMSに頼らず、必要な編集体験を自分で組み立てた点が工夫である。

### (4) 権限分離（管理者のみ編集可能）🔒

閲覧は誰でも可能だが、**記事の作成・編集・削除はJWT認証を通った管理者のみ**に限定している。フロントではルーティングと表示を制御し、**バックエンド側でも認証ガード（`jwt-auth.guard`）で保護**しているため、APIを直接叩かれても権限のない操作はできない。「フロントで隠すだけ」にせず、サーバ側で実際の認可を行っている点を意識した。

本作品はWebアプリケーションでゲームとは領域が異なるが、以下は職種を問わず活かせる強みだと考えている。

## 8. 動作確認方法

### 公開サイトで確認する

1. https://mocaluna0117.github.io/ds-club-web にアクセス
2. メンバー紹介・ブログ・活動記録・お問い合わせを閲覧/操作
   - ※APIが起動していれば1秒以内に表示される（keep-warm運用中）

### ローカルで動かす場合

```bash
# 前提: Node.js 20+, Docker

# DB（PostgreSQL）を起動
docker compose up -d

# バックエンド
cd backend && npm install && npx prisma migrate dev && npm run start:dev

# フロントエンド
cd frontend && npm install && npm run dev
```
